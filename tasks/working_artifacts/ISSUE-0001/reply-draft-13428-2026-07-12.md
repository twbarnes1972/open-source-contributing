Thanks — that framing makes sense, and I wasn't expecting checkout exclusivity to be on the
table given that brief sequential overlap on a StaticPool is legitimate when transactions
don't actually interleave.

If docs are the right fix, I'm happy to do the work: I'd propose a warning in the three
places that recommend or auto-select StaticPool for in-memory SQLite —

1. the aiosqlite dialect docs (where `:memory:` URLs get StaticPool automatically),
2. the pysqlite "memory database in multiple threads" section, and
3. the `StaticPool` docstring itself (it currently advertises asyncio compatibility with no
   concurrency caveat) —

each stating that StaticPool provides a single connection with no checkout exclusivity, so
sessions open concurrently share one transaction scope: a `close()`/rollback in one can
silently discard another's flushed-but-uncommitted work. Guidance: keep session lifetimes
non-overlapping, or use a file-backed database when concurrent sessions are needed (that also
covers `cache=shared` named-memory URIs, which lose rows the same way).

Two small questions before I push a change through Gerrit:

- Would you want the docs to point at `AssertionPool` as an opt-in way for test suites to
  *detect* accidental overlap (it raises on the second checkout, which would have surfaced
  this immediately for us)? It exists already; it just isn't visible in any of the memory-DB
  sections.
- Is there any appetite for a warn-not-raise on concurrent checkout of StaticPool's sole
  connection (off by default or opt-in)? If that's a non-starter, no problem — docs alone
  close the trap for the next person landing on #6987.
