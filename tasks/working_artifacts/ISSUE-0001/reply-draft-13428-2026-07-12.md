Thanks — that framing makes sense. Rather than argue hypotheticals I spent some time
implementing and testing options against `main`, and found something that changes the docs
story: **the memdb VFS already works end-to-end through both sqlite dialects today.**

```python
engine = create_engine("sqlite:///file:/shared_db?vfs=memdb&uri=true")
# aiosqlite equivalent works too; both select the file-db pool (QueuePool /
# AsyncAdaptedQueuePool), so every checkout is its own connection to one
# shared in-memory database
```

With that URL form, the exact scenario from my report — session B's close-rollback
destroying session A's flushed-but-uncommitted INSERT — can't occur: each session has its own
connection and transaction. Re-running my reproducer against it, the previously-lost row
survives, and contention surfaces as a loud `database is locked` instead of silent loss.
Caveats I hit while validating (all worth documenting): the name must start with a slash
(`file:/name` — a relative name gives each connection a *private* memory db, no sharing);
no WAL on memdb, so a pending write blocks readers on other connections until `timeout`;
the database lives only while at least one connection is open; requires SQLite 3.36+.

So for the docs I'd propose, and am happy to submit via Gerrit:

1. A warning in the three places that recommend/auto-select StaticPool for memory DBs
   (pysqlite memory-db section, aiosqlite pooling section, StaticPool docstring) stating
   that transactional state is shared across concurrent checkouts and sessions must not
   overlap.
2. A short "sharing a memory database across connections" section presenting the memdb URL
   form as the supported pattern for concurrent sessions, with the caveats above. I have
   both written, plus tests (pool-class selection for memdb URLs + a two-connection sharing
   round-trip gated on SQLite 3.36+).

On detection: I also prototyped a warning in StaticPool itself — best-effort via the
driver's `in_transaction`, firing (a) when the sole connection is checked out while a
transaction begun via another still-open checkout is in progress, and (b) when
rollback-on-return would discard an in-progress transaction while other checkouts remain
open. It's warning-only, zero false positives across the pool suite, and catches the
canonical orderings of this bug. Honest limitation found while testing: rollbacks that the
Connection/Session layer issues itself (e.g. session close) happen before the pool sees
anything, so pool-level detection is inherently partial — which I think supports your point
that real enforcement can't live there cleanly. Happy to include or drop that part; the
docs + memdb guidance are the substance.

Would you take these through Gerrit — docs only, or docs + warning?
