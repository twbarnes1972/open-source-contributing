Thanks both. Since the docs section is being rewritten right now, here's validated data
that may be useful for it — I re-ran my reproducer against the alternatives on current main.

**Your `cache=shared` recipe works — and I need to correct my original post.** I wrote that
`cache=shared` named-memory URIs "also lose rows"; that turns out to be a URL-spelling
artifact, not a property of shared cache:

- `sqlite+pysqlite:///file::memory:?cache=shared&uri=true` (your spelling) → classified as
  file-db → QueuePool → **no loss** (verified sync and aiosqlite: the flushed row survives,
  the concurrent reader gets an immediate `database table is locked`).
- `sqlite+aiosqlite:///file:name?mode=memory&cache=shared&uri=true` (the spelling I had
  tested) → `_is_url_file_db()` sees `mode=memory` → **StaticPool** (SingletonThreadPool on
  pysqlite) → back on one shared connection, rows lost exactly as in my report. Forcing
  `poolclass=QueuePool` on the same URL: no loss.

So the data loss follows the pool in every configuration; shared cache itself is fine. Two
equivalent SQLite spellings of "named shared memory db" currently get different pools, and
the `mode=memory` one lands on the silently-unsafe side. Maybe worth either treating
`cache=shared` as multi-connection-capable in the pool-class selection, or calling out the
spelling distinction in the rewritten docs.

**The memdb VFS also checks out** as a modern alternative (SQLite's own docs discourage
shared-cache and offer memdb as its replacement for this use case):

    engine = create_engine("sqlite:///file:/shared_db?vfs=memdb&uri=true")

Already classifies as file-db on main, sync and aiosqlite. Same no-silent-loss behavior.
Caveats verified: the name must start with a slash (`file:/name` — relative names give each
connection a *private* db, no error); no WAL, so a pending write blocks readers on other
connections (but via SQLITE_BUSY, so `timeout` applies — shared-cache table locks fail
immediately); db lifetime = at least one open connection; needs SQLite 3.36+.

I have a docs draft covering the above plus tests (pool-class selection for memdb URLs, a
two-connection sharing round-trip gated on 3.36+) on a local branch — happy to push any of
it through Gerrit, or just leave this here for the rewrite to draw from, whichever is less
friction.

@CaselIT on detection: I prototyped a warning in StaticPool (fires when the sole connection
is checked out while a transaction begun via another still-open checkout is in progress, and
when rollback-on-return would discard an in-progress transaction with other checkouts open —
keyed on the driver's `in_transaction`, warning-only, no false positives across the pool
suite). Honest finding: rollbacks the Session/Connection layer issues happen before the pool
sees anything, so pool-level detection is inherently partial — which I think confirms your
instinct that enforcement doesn't fit there cleanly. Available if wanted; the docs are the
substance.
